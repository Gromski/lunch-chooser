# Environment File Configuration Note

## Important: `.env` vs `.env.local`

### For Next.js App (Runtime)
- Next.js automatically loads `.env.local`
- This is what your application uses when running (`npm run dev`)
- `.env.local` is already in `.gitignore` (secure)

### For Prisma CLI (Migrations)
- Prisma CLI looks for `.env` by default
- You need both `.env` and `.env.local` with the same values
- Both files are in `.gitignore` (secure)

## Setup

When you create your environment variables:

```bash
# Create .env.local for Next.js
cp .env.example .env.local

# Create .env for Prisma CLI
cp .env.local .env

# Or do both at once:
cp .env.example .env.local && cp .env.local .env
```

## Keeping in Sync

Whenever you update environment variables:
1. Update `.env.local` first
2. Copy changes to `.env`: `cp .env.local .env`

## Why Both Files?

- **`.env.local`** - Next.js convention, loaded by Next.js automatically
- **`.env`** - Standard convention, loaded by most CLI tools (Prisma, etc.)

Both files contain the same values and both are gitignored for security.

## Alternative Approach

If you prefer to use only `.env.local`, you can install `dotenv-cli`:

```bash
npm install -D dotenv-cli
```

Then update `package.json` scripts:
```json
"db:migrate": "dotenv -e .env.local -- prisma migrate dev",
"db:generate": "dotenv -e .env.local -- prisma generate",
```

But the dual-file approach is simpler for most cases.
